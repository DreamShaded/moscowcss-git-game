# Reflog — журнал всех движений HEAD

Reflog — это локальный журнал, куда git записывает каждое изменение положения HEAD и веток: коммиты, переключения, reset, rebase, merge. По умолчанию записи хранятся 90 дней. Если ты "потерял" коммиты после reset --hard или случайного удаления ветки — reflog почти наверняка поможет их найти.

Reflog хранится в `.git/logs/`. Он **локальный** — у каждого разработчика свой, push его не передаёт.

## Когда это нужно

- После `git reset --hard HEAD~5` вернуть потерянные коммиты.
- Найти коммит, который был на удалённой ветке до её удаления.
- Понять, что вообще происходило в репо: какие команды запускались и когда.
- Восстановить ветку после `git branch -D`.

## Как использовать

```bash
git reflog                        # история HEAD (последние операции сверху)
git reflog show main              # история конкретной ветки
git reflog --date=iso             # с человекочитаемыми датами

# Восстановление: найти нужный хеш в reflog, потом
git reset --hard <hash>           # переместить ветку на тот коммит
# или
git branch recovery <hash>        # создать новую ветку из того коммита
```

Формат вывода reflog: `<hash> HEAD@{n}: <действие>: <описание>`. Индекс `n` — сколько шагов назад.

## Подвохи

- **Reflog — только локальный.** Если ты клонировал репо и сразу что-то потерял — reflog поможет только если ты уже успел поработать. На чужом компьютере своего reflog нет.
- **Хранится 90 дней** по умолчанию (`gc.reflogExpire`). После этого gc может удалить старые записи. Для недостижимых коммитов срок меньше — 30 дней (`gc.reflogExpireUnreachable`).
- **`git reflog` и `git log` — разное.** `git log` показывает историю коммитов дерева. Reflog показывает историю *движений* HEAD — включая переключения между ветками, которые в `git log` не видны.
- **После `git gc`** недостижимые коммиты могут быть удалены. Восстановить после этого сложнее — только через `git fsck --lost-found`.
