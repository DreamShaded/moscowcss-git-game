# Восстановление после неудачного rebase

Ребейз пошёл не так — история запуталась, конфликты не разрешаются, или ты просто передумал. Есть несколько способов выйти из ситуации в зависимости от того, на каком этапе всё пошло не так.

## Если rebase ещё не завершён (в середине)

```bash
git rebase --abort               # вернуть всё как было до git rebase
```

`--abort` полностью откатывает ребейз: ветка возвращается на позицию до старта. Работает пока `.git/rebase-merge/` или `.git/rebase-apply/` ещё существуют.

## Если rebase уже завершился, но результат плохой

Git устанавливает `ORIG_HEAD` перед стартом ребейза:

```bash
git reset --hard ORIG_HEAD       # откатить ветку на позицию до rebase
```

Это работает, если после ребейза не было других опасных операций (иначе ORIG_HEAD перезаписан).

## Если ORIG_HEAD уже перезаписан

Используй reflog:

```bash
git reflog                       # найти запись вида "rebase finished" или последний коммит до rebase
git reset --hard <hash>          # откатиться на нужную позицию
```

Ищи в reflog запись типа `HEAD@{n}: rebase finished:` — хеш *перед* этой записью и есть то, что было до ребейза.

## Если ребейз создал дубли коммитов

После ребейза старые коммиты (до ребейза) становятся dangling — они в объектной базе, но ни на что не ссылаются. Через reflog их ещё можно найти. Через gc — нет.

## Подвохи

- **`git rebase --abort` в interactive rebase** тоже работает — неважно, на каком шаге ты остановился.
- **После `git rebase --continue`** ребейз продолжается с текущего конфликтного коммита. Если ты сделал `continue` и потом понял что ошибся — `--abort` уже недоступен (ребейз завершён). Нужен ORIG_HEAD или reflog.
- **Rebase в shared-ветке** создаёт дополнительную проблему: даже если ты откатился локально, коллеги уже могли спулить ребейзнутую историю. Нужен [`post-rewrite-team-protocol`](./post-rewrite-team-protocol.md).
