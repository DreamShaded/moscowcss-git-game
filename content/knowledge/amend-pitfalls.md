# Подвохи `git commit --amend`

`--amend` — удобная команда, но у неё есть несколько неочевидных последствий, о которых стоит знать до того, как они укусят.

## Хеш коммита всегда меняется

`--amend` создаёт **новый объект коммита**, даже если ты не изменил ни одного файла и не поменял сообщение. Хеш определяется содержимым, автором, датой и родительским хешем — `--amend` создаёт новый объект с текущим временем.

Это значит: всё, что ссылалось на старый хеш (теги, другие ветки, CI-ссылки в комментариях PR), теперь ссылается на "потерянный" коммит.

## Git hooks могут не сработать

Некоторые команды при `--amend` не перезапускают все hooks. В частности, `post-commit` hook запускается, но `commit-msg` hook тоже запускается (с новым сообщением). Если твои hooks проверяют что-то специфичное для первого коммита — убедись, что они корректно работают при amend.

## Автор и дата

```bash
git commit --amend --reset-author   # обновить автора на текущего git user
git commit --amend --date=now       # обновить дату на текущую
```

По умолчанию `--amend` сохраняет оригинального автора и дату коммита, но создаёт новую дату committer. При `--reset-author` оба обновляются.

## Нельзя амендить не-HEAD

`--amend` работает только с последним коммитом (HEAD). Для исправления более ранних коммитов — `git rebase -i` с командой `edit` или `reword`.

## После push — force push обязателен

После `--amend` на уже запушенном коммите нужен `--force-with-lease`. Подробнее: [`amend-after-push`](./amend-after-push.md).
