# Cherry-pick merge-коммитов — флаг `-m`

Обычный `git cherry-pick <merge-hash>` завершится ошибкой: "is a merge but no -m option was given". Merge-коммит имеет двух родителей, и git не знает, относительно какого из них считать "изменение".

Флаг `-m N` (mainline) указывает, какой родитель считать основной линией: изменения вычисляются как diff от этого родителя до merge-коммита.

## Как использовать

```bash
# Перенести merge-коммит, считая первого родителя основным:
git cherry-pick -m 1 <merge-hash>

# Перенести merge-коммит, считая второго родителя основным:
git cherry-pick -m 2 <merge-hash>
```

**Родитель 1** (`-m 1`) — ветка, в которую мержили (обычно `main` или `develop`). Это самый частый выбор: ты переносишь "что привнесла влитая ветка".

**Родитель 2** (`-m 2`) — ветка, которую мержили. Используется редко.

## Как узнать, кто первый родитель

```bash
git show <merge-hash>       # в строке Merge: первый хеш — родитель 1, второй — родитель 2
git log --merges --parents  # история merge-коммитов с хешами родителей
```

## Когда это нужно

- Backport: перенести merge feature-ветки в stable/release.
- Hotfix в нескольких ветках: оригинальный fix был смержен в develop, нужно перенести в main.

## Подвохи

- **Cherry-pick merge-коммита — не то же самое, что cherry-pick всей ветки.** Merge-коммит несёт только diff "что изменилось при слиянии". Если в влитой ветке было 10 коммитов, cherry-pick merge-коммита перенесёт суммарное изменение одним коммитом.
- **Конфликты часты** при переносе merge-коммитов: контекст другой ветки может не совпадать.
- **Вместо cherry-pick merge** нередко лучше cherry-pick отдельных коммитов из влитой ветки или `git rebase --onto`.
