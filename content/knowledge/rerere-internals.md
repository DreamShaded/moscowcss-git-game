# `git rerere` — автоматическое повторное разрешение конфликтов

`rerere` расшифровывается как "reuse recorded resolution" — повторное использование записанного разрешения. Если ты разрешаешь один и тот же конфликт несколько раз (например, каждый раз при мерже `main` в долгоживущую ветку), rerere запомнит решение и применит его автоматически.

## Как это работает

Git хранит "конфликтные паттерны" в `.git/rr-cache/`. Когда rerere включён:

1. При возникновении конфликта git записывает pre-image (файл с маркерами конфликта) в `.git/rr-cache/<hash>/preimage`.
2. Когда ты разрешаешь конфликт и делаешь `git add` — git записывает post-image (результат разрешения).
3. В следующий раз при таком же конфликте git автоматически применяет сохранённое разрешение.

## Как включить и использовать

```bash
# Включить rerere (рекомендуется глобально, но работает и per-repo)
git config --global rerere.enabled true
git config rerere.enabled true       # только для текущего репо

# Посмотреть статус записанных разрешений
git rerere status                    # файлы, для которых есть записанное решение
git rerere diff                      # разница: что было записано vs что сейчас

# После автоматического разрешения проверь результат!
git diff                             # убедись что всё применилось правильно
git add <file>
git commit
```

## Подвохи

- **Rerere не гарантирует правильность.** Он механически воспроизводит прошлое решение. Если контекст изменился, а конфликтные маркеры совпали — применится устаревшее решение. Всегда проверяй `git diff` после автоматического разрешения.
- **Ключ кэша — хеш конфликтного паттерна**, не имя файла. Один и тот же файл с разными конфликтами — разные записи.
- **Если записал неправильное решение** — `git rerere forget <file>` удаляет запись. Подробнее: [`rerere-forget`](./rerere-forget.md).
- **`--global` рекомендуется**, но не обязателен. Per-repo включение (`git config rerere.enabled true` без `--global`) работает так же, только для конкретного репо.
