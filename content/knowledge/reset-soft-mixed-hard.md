# Три режима `git reset`: --soft, --mixed, --hard

`git reset` перемещает указатель ветки на другой коммит. Режим определяет, что при этом происходит с индексом (staging area) и рабочей директорией (worktree).

## Три режима наглядно

| Режим | Ветка (HEAD) | Индекс | Worktree |
|-------|-------------|--------|---------|
| `--soft` | перемещается | не меняется | не меняется |
| `--mixed` (по умолчанию) | перемещается | сбрасывается | не меняется |
| `--hard` | перемещается | сбрасывается | сбрасывается |

**`--soft`**: коммиты "распаковываются" обратно в индекс. Изменения готовы к новому коммиту. Удобно для squash: `git reset --soft HEAD~3` → `git commit -m "..."`.

**`--mixed`**: коммиты распакованы в worktree, но не в индекс. Изменения видны в `git status` как неиндексированные. Это поведение по умолчанию — `git reset HEAD~1` делает именно это.

**`--hard`**: коммиты удалены из истории, изменения из worktree тоже выброшены. Необратимо для незакоммиченных изменений — даже reflog не поможет восстановить то, что не было закоммичено.

## Как использовать

```bash
git reset --soft HEAD~3          # схлопнуть 3 коммита в один (изменения в индексе)
git reset HEAD~1                 # "отменить" последний commit (изменения остаются в файлах)
git reset --hard HEAD~1          # выбросить последний коммит вместе с изменениями
git reset --hard ORIG_HEAD       # откатить после неудачного merge/rebase
git reset --hard origin/main     # привести ветку в соответствие с remote
```

## Подвохи

- **`--hard` необратим для незакоммиченного.** Сами коммиты можно восстановить через reflog, но незакоммиченные изменения из worktree пропадают навсегда.
- **`git reset` в shared-ветке — плохая идея.** Если ветку уже спулили другие — reset + force-push сломает их историю. Используй `git revert` для публичных веток.
- **`--mixed` — это дефолт.** Если просто написать `git reset HEAD~1` без флага — будет `--mixed`.
- **Reset не трогает stash.** Stash живёт отдельно и не затрагивается ни одним из режимов.
