# Cherry-pick диапазонов коммитов

`git cherry-pick` умеет переносить не только один коммит, но и диапазоны. Синтаксис диапазонов тут немного отличается от `git log` — важно понимать разницу между `A..B` и `A^..B`.

## Синтаксис диапазонов

**`git cherry-pick A..B`** — переносит коммиты от A (не включая A) до B (включая B). Коммит A не копируется.

**`git cherry-pick A^..B`** — переносит коммиты от A (включая A) до B (включая B). Это чаще то, что нужно когда хочешь перенести "с коммита A по коммит B включительно".

```bash
# Перенести 3 последних коммита из feature/grid:
git cherry-pick HEAD~3..HEAD    # не включает HEAD~3
git cherry-pick HEAD~3^..HEAD   # включает HEAD~3 (то есть 4 коммита — осторожно)

# Перенести диапазон по хешам:
git cherry-pick abc123..def456   # не включая abc123
git cherry-pick abc123^..def456  # включая abc123
```

## Флаги для диапазонов

```bash
git cherry-pick -n A..B          # --no-commit: применить изменения без создания коммитов
git cherry-pick -x A..B          # добавить "(cherry picked from commit <hash>)" к сообщению
```

## Когда это нужно

- Перенести серию связанных коммитов из feature-ветки в hotfix.
- Backport набора фиксов в stable-ветку.
- Перенести работу из одного форка в другой.

## Подвохи

- **Порядок важен.** Cherry-pick диапазона применяет коммиты в хронологическом порядке (от старого к новому). Если коммиты зависят друг от друга — это правильно. Если независимы — можно переносить выборочно.
- **Конфликты на каждом коммите** разрешаются отдельно. При конфликте на третьем из пяти коммитов: разрешить → `git add` → `git cherry-pick --continue`.
- **`A..B` не включает A** — типичная ошибка. Если хочешь включить оба конца — `A^..B` или явный список хешей.
- **Merge-коммиты в диапазоне** требуют флага `-m`. Подробнее: [`cherry-pick-merge-commits`](./cherry-pick-merge-commits.md).
