# Exit codes для `git bisect run`

`git bisect run <command>` автоматизирует бинарный поиск: git запускает команду на каждом коммите и по exit code определяет, хороший коммит или плохой. Важно понимать семантику кодов — неправильный код сломает весь поиск.

## Таблица exit codes

| Exit code | Значение для bisect | Пример |
|-----------|-------------------|--------|
| `0` | Хороший коммит (good) | тесты прошли |
| `1–127` (кроме 125) | Плохой коммит (bad) | тесты упали |
| `125` | Пропустить коммит (skip) | коммит не собирается |
| `128+` | Ошибка bisect, прерывание | скрипт упал неожиданно |

## Примеры скриптов

```bash
# Простейший: npm-скрипт возвращает 0/1 сам по себе
git bisect run npm run test:hover

# Bash-скрипт с явными кодами
git bisect run bash -c '
  npm run build || exit 125   # не собирается — пропускаем
  npm run test:hover          # 0 = good, иначе bad
'

# Проверка наличия строки в файле
git bisect run grep -q "broken-class" src/styles/button.css
# grep: 0 = нашёл, 1 = не нашёл — тут "нашёл" = плохой коммит
# Нужно инвертировать: git bisect run bash -c '! grep -q "broken-class" ...'
```

## Подвохи

- **`125` — специальный код для skip**, не просто "ошибка". Используй его когда коммит нельзя проверить по не связанной причине (не собирается, тест нерелевантен).
- **Код 126 и 127** — "команда не найдена" или "нет прав". Git интерпретирует их как "плохой коммит", что некорректно. Убедись, что скрипт найден и исполняем.
- **Код 128+** прерывает bisect полностью. Если скрипт падает с сегфолтом или убивается сигналом — bisect останавливается с ошибкой.
- **`git bisect reset` обязателен** после завершения — даже если bisect run нашёл результат. Репо остаётся в состоянии bisecting.
- **Скрипт запускается в detached HEAD**. Не делай в нём коммиты — они потеряются.
