# gc и pruning — когда git удаляет dangling-коммиты

`git gc` (garbage collect) — команда, которая чистит объектную базу git: упаковывает объекты в pack-файлы, удаляет недостижимые объекты, обрезает старые записи reflog. Она запускается автоматически в фоне после некоторых операций.

Dangling-объекты (коммиты, деревья, блобы) — это объекты, на которые больше не ссылается ни одна ветка, тег или reflog-запись. Они появляются после `git reset --hard`, `git branch -D`, ребейза.

## Когда это важно

Если ты потерял коммиты через `reset --hard` или удаление ветки и хочешь их восстановить — у тебя есть окно времени до следующего gc. По умолчанию:

- Достижимые reflog-записи хранятся **90 дней** (`gc.reflogExpire`)
- Недостижимые объекты хранятся **30 дней** (`gc.reflogExpireUnreachable`)

После этого gc может удалить их окончательно.

## Как работает

```bash
git gc                           # запустить сборку мусора вручную
git gc --prune=now               # удалить недостижимые объекты немедленно
git gc --auto                    # запустить только если накопилось достаточно мусора

git fsck --unreachable           # показать недостижимые объекты (до gc)
git prune --dry-run              # показать, что было бы удалено
```

`git gc --auto` запускается автоматически после `git commit`, `git fetch`, `git merge` и других команд, когда количество loose-объектов превышает порог (`gc.auto`, по умолчанию 6700).

## Подвохи

- **После `gc --prune=now`** dangling-коммиты теряются навсегда. Даже `git fsck` их не найдёт.
- **Reflog защищает от gc.** Пока запись в reflog указывает на коммит, gc его не удалит. Именно поэтому 30/90 дней — реальная защита.
- **`git remote prune origin`** — другая операция, удаляет локальные remote-tracking ветки, которых больше нет на remote. Не путать с gc.
- **В воркспейсах с worktree** gc работает аккуратнее: объекты, достижимые из любого worktree, не удаляются.
