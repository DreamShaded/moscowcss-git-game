# `log -S`, `log -G`, `log --grep` — три разных поиска по истории

Все три флага фильтруют коммиты в `git log`, но ищут разное. Путаница между ними — частая причина "почему git ничего не нашёл".

## Разница

**`git log -S "строка"`** (pickaxe) — находит коммиты, в которых **количество вхождений** строки изменилось. Если строка добавлена или удалена — коммит в результатах. Если строка была и осталась (просто переместилась) — не попадает.

**`git log -G "regex"`** — находит коммиты, где в **diff** есть хотя бы одна строка, соответствующая регулярному выражению. Более широкий: ловит и добавление, и удаление, и коммиты где строка просто рядом с изменением.

**`git log --grep "текст"`** — ищет по **сообщению коммита**, не по содержимому файлов. Совсем другая семантика.

## Примеры

```bash
# Найти коммиты, где добавили или убрали !important в button.css:
git log -S "!important" -- styles/button.css

# Найти коммиты, где diff содержит строки с z-index:
git log -G "z-index:\s*\d+" -- styles/

# Найти коммиты с "hotfix" в сообщении:
git log --grep "hotfix"

# Комбинация: поиск по содержимому + конкретный автор:
git log -S "color: red" --author="Рома" -- styles/
```

## Когда что использовать

- **`-S`**: "когда эта функция/свойство появилась или исчезла?" — точный ответ.
- **`-G`**: "в каких коммитах трогали эту область кода?" — более широкий поиск.
- **`--grep`**: "найти коммиты по описанию" — для поиска по истории задач/тикетов.

## Подвохи

- **`-S` не ловит рефакторинг.** Если `!important` переехал из одной строки в другую в рамках одного коммита (количество не изменилось) — `-S` этот коммит пропустит. `-G` поймает.
- **`-G` может давать ложные срабатывания.** Коммит попадает если паттерн есть в любой строке diff — даже если сама искомая строка не менялась, а рядом добавили комментарий.
- **`--`** перед путём отделяет пути от ревизий: `git log -S "text" -- path/to/file`.
