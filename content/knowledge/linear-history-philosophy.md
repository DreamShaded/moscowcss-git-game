# Линейная история — зачем команды её любят

Линейная история — это когда `git log` выглядит как прямая цепочка коммитов без разветвлений и merge-коммитов. Достигается через rebase вместо merge при интеграции веток.

## Почему это важно

**Читаемость.** `git log --oneline` превращается в понятную последовательность событий. В нелинейной истории — запутанный граф слияний, особенно при нескольких параллельных ветках.

**Bisect работает лучше.** При линейной истории `git bisect` однозначно проходит по всем коммитам. В нелинейной — merge-коммиты путают bisect, иногда нужны дополнительные флаги.

**`git log -S`, `git blame`** дают точные результаты. Merge-коммиты могут маскировать реальный коммит с изменением.

## Rebase вместо merge для feature-веток

```bash
# Вместо git merge main:
git fetch origin
git rebase origin/main        # перенести коммиты поверх актуального main

# Или через pull:
git pull --rebase origin main
```

После ребейза история выглядит как будто ты начал работу от самого свежего main — даже если реально ветвился неделю назад.

## Когда merge всё-таки уместен

- **Долгоживущие feature-ветки** с несколькими авторами — rebase затрудняет совместную работу.
- **Когда важен факт слияния** — merge-коммит явно фиксирует "здесь feature X вошла в main".
- **Публичные ветки** — rebase меняет хеши, что неудобно для участников.

## `--ff-only` как страховка

```bash
git merge --ff-only feature/grid
```

Разрешает merge только если возможно fast-forward (без merge-коммита). Если нельзя — ошибка. Используется в CI для проверки что ветка ребейзнута на актуальный main.

## Подвохи

- **Rebase переписывает историю.** После `git rebase` хеши коммитов меняются — нужен force push.
- **Squash-merge** (GitHub: "Squash and merge") — компромисс: один коммит вместо всей feature-ветки. История линейная, но теряются отдельные шаги разработки.
- **Линейная история — это договорённость**, а не техническое требование. Важно чтобы вся команда следовала одному подходу.
