# `git submodule update` vs `update --remote` — критическая разница

Это одна из самых частых ловушек с сабмодулями. Два похожих вызова делают принципиально разные вещи.

## Разница

**`git submodule update`** (без `--remote`) — checkout сабмодуля на коммит, записанный в родительском репо. Детерминированный, воспроизводимый, "как зафиксировано".

**`git submodule update --remote`** — скачивает последние изменения из upstream сабмодуля и переключается на самый свежий коммит ветки, игнорируя записанный pointer. Это обновляет сабмодуль до HEAD его remote-ветки.

## Когда что использовать

```bash
# Синхронизировать с зафиксированным состоянием (обычная работа, CI):
git submodule update --init --recursive

# Намеренно обновить сабмодуль до последней версии:
git submodule update --remote design-tokens
# После этого ОБЯЗАТЕЛЬНО зафиксировать новый pointer:
git add design-tokens
git commit -m "chore: bump design-tokens to latest"
```

## Почему `--remote` опасен без понимания

Если запустить `--remote` думая что это "просто обновление" — ты:
1. Получаешь в сабмодуле новый коммит, которого нет в зафиксированном pointer'е.
2. Если сделать коммит — в репо попадёт новый pointer, который все остальные получат при `git pull`.
3. Если не сделать коммит — у тебя "грязный" сабмодуль, который сбивает с толку `git status`.
4. Если этот новый коммит есть только локально в сабмодуле и не запушен — CI упадёт.

## Подвохи

- **`--remote` без явного имени сабмодуля** обновит все сабмодули до их remote HEAD. Это редко то, что нужно.
- **Ветка для `--remote`** берётся из `.gitmodules` (поле `branch`) или из HEAD сабмодуля. По умолчанию — `main` или `master`.
- **В задаче H6** (`git submodule update --remote` в forbidden-командах) именно потому: ты хочешь откатиться на предыдущий pointer, а не взять новейший из remote.
