# `git fsck --lost-found` — восстановление потерянных коммитов

`git fsck` проверяет целостность объектной базы git и находит объекты, на которые больше никто не ссылается (dangling). `--lost-found` записывает их в `.git/lost-found/` для ручного осмотра.

Это последний рубеж восстановления: когда reflog уже не помогает (ветка удалена, ORIG_HEAD перезаписан), но gc ещё не успел удалить объекты.

## Когда это нужно

- Удалили ветку `git branch -D`, потом `git push origin --delete`, и только потом вспомнили, что там была неотмеченная работа.
- Сделали `git reset --hard`, потом ещё несколько операций — ORIG_HEAD перезаписан, reflog запутан.
- Нашли странный хеш в чужом комментарии к PR, хотите посмотреть что в нём.

## Как использовать

```bash
git fsck --lost-found            # найти dangling-объекты, записать в .git/lost-found/
git fsck --unreachable           # показать недостижимые объекты (без записи)
git fsck --dangling              # показать только dangling (не просто unreachable)

# Осмотр найденного:
ls .git/lost-found/commit/       # список dangling-коммитов
git show <hash>                  # посмотреть конкретный коммит
git log <hash>                   # посмотреть историю от этого коммита

# Восстановление:
git branch recovered <hash>      # создать ветку на найденном коммите
```

## Подвохи

- **Разница между dangling и unreachable.** Dangling — объект не достижим вообще ни из чего. Unreachable — не достижим из именованных веток и тегов, но может быть в reflog. `--lost-found` работает с dangling.
- **Это не работает после gc --prune.** Если gc уже удалил объекты — fsck их не найдёт. Окно восстановления — 30 дней после потери (для недостижимых).
- **Много шума.** `git fsck` часто выводит предупреждения о "broken links" и "missing blob" в stash или worktree-временных объектах. Это нормально, не паникуй.
- **`.git/lost-found/`** содержит только голые хеши без имён. Придётся вручную смотреть каждый через `git show`.
