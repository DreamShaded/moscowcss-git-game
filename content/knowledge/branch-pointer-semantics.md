# Ветка — указатель на коммит, не контейнер

В git ветка — это просто файл с хешем коммита. Буквально: `.git/refs/heads/main` содержит одну строку — SHA-1 последнего коммита. Никакого "содержимого ветки" не существует. Это важно понять, чтобы не удивляться поведению многих команд.

## Что это означает на практике

Когда ты делаешь коммит — git создаёт объект коммита, записывает его хеш в файл текущей ветки. Ветка "переехала" на новый коммит. Старый коммит никуда не делся — он просто больше не является последним.

Когда ты делаешь `git branch feature` — git просто копирует текущий хеш в новый файл `.git/refs/heads/feature`. Никакие файлы не копируются, история не дублируется.

Когда ты делаешь `git reset --hard HEAD~1` — git записывает хеш родителя в файл ветки. Коммит не удаляется из объектной базы git сразу — он становится "недостижимым" и удалится только через gc.

## Подвохи

- **Удалить ветку — значит удалить только указатель.** Коммиты остаются в объектной базе ещё 30–90 дней. Через reflog их можно найти.
- **Merge не перемещает коммиты.** При `git merge feature` в `main` git создаёт новый merge-коммит, который ссылается на оба родителя. Коммиты из `feature` физически не "переходят" в `main` — они просто становятся достижимы через `main`.
- **Rebase создаёт новые коммиты.** При ребейзе git создаёт копии коммитов с новыми родителями и новыми хешами. Старые коммиты остаются в объектной базе (до gc).
- **`git branch -d` требует merged.** Git проверяет, достижимы ли коммиты ветки из других веток. Если нет — `-d` откажет, нужен `-D`. Это защита от потери работы.
