# Автоматические refs: ORIG_HEAD, MERGE_HEAD, FETCH_HEAD, HEAD@{n}

Git сам поддерживает несколько зарезервированных имён, которые обновляются в процессе работы. Это не настройки — это живые указатели на коммиты, которые можно использовать в командах вместо хешей.

## Что есть что

**`ORIG_HEAD`** — позиция ветки *до* последней опасной операции (`reset`, `merge`, `rebase`, `pull`, `cherry-pick`). Лежит в `.git/ORIG_HEAD`. Перезаписывается при каждой новой опасной операции.

**`MERGE_HEAD`** — хеш коммита, который сейчас вливается в ветку. Существует только во время незавершённого merge. Git создаёт его при старте, удаляет после `commit` или `--abort`.

**`FETCH_HEAD`** — хеш (или несколько) того, что было скачано последним `git fetch`. Используется командами типа `git merge FETCH_HEAD` в старом стиле или просто для информации.

**`HEAD@{n}`** — позиция HEAD n шагов назад в истории reflog. `HEAD@{1}` — предыдущее положение HEAD, `HEAD@{2}` — до него. Меняется при каждом переключении ветки, reset, commit.

## Когда это нужно

```bash
git reset --hard ORIG_HEAD      # откатить последний reset/merge/rebase
git show MERGE_HEAD             # посмотреть коммит, который вливаем
git log HEAD..FETCH_HEAD        # что fetch принёс нового
git reset --hard HEAD@{1}       # альтернатива ORIG_HEAD через reflog
```

`MERGE_HEAD` полезен при разрешении конфликтов — `git diff HEAD MERGE_HEAD` покажет разницу между двумя сторонами слияния.

## Подвохи

- **`ORIG_HEAD` перезаписывается** при каждой опасной операции. Если нужно откатиться на два шага назад — reflog.
- **`MERGE_HEAD` исчезает** после завершения или прерывания merge. После `git merge --abort` он уже не доступен.
- **`HEAD@{n}` — история HEAD**, не ветки. Если ты переключился между ветками, `HEAD@{1}` укажет на коммит предыдущей ветки, а не предыдущий коммит текущей. Для истории конкретной ветки — `git reflog show <branch>`.
