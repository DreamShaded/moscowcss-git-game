# Когда git запускает gc — и что это значит для восстановления

`git gc --auto` запускается автоматически после ряда команд. Это нормальное поведение, но в контексте восстановления потерянных коммитов важно понимать, когда именно это происходит — и когда оно ещё не произошло.

## Когда запускается auto gc

Git проверяет условие auto gc после:
- `git commit`
- `git fetch` / `git pull`
- `git merge`
- `git rebase`
- `git am`

Условие: количество loose-объектов превышает `gc.auto` (по умолчанию 6700) или количество pack-файлов превышает `gc.autoPackLimit` (по умолчанию 50).

При нормальной работе в небольших репо auto gc запускается редко — раз в несколько недель или месяцев.

## Что удаляется при gc

- **Недостижимые объекты старше 2 недель** (параметр `--prune=2.weeks.ago` по умолчанию)
- **Reflog-записи старше 90 дней** для достижимых объектов
- **Reflog-записи старше 30 дней** для недостижимых объектов

Всё, что моложе этих порогов, gc не трогает — даже если объект недостижим.

## Практически

```bash
# Проверить, когда последний раз запускался gc
git reflog --date=iso | tail -5  # смотрим дату старых записей

# Принудительно отложить gc (если нужно восстановить что-то)
git config gc.auto 0             # отключить auto gc
# ... сделать восстановление ...
git config --unset gc.auto       # вернуть настройку

# Принудительный запуск
git gc
```

## Подвохи

- **После `git rebase`** gc запускается автоматически. Если ребейз прошёл плохо и ты сделал ещё несколько операций — окно для восстановления через `fsck` может сузиться.
- **`--prune=now`** в ручном gc удаляет немедленно, без ожидания двух недель. Не используй без понимания последствий.
- **GitHub и другие хостинги** запускают gc на своих серверах независимо от тебя. Remote reflog недоступен пользователям. Если коммит не запушен — remote не поможет.
