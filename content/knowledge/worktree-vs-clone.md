# Worktree vs Clone — когда что выбирать

И worktree, и clone позволяют работать с репозиторием в нескольких местах одновременно. Разница — в том, как они связаны с исходным репо и какой ценой.

## Сравнение

| | `git worktree add` | `git clone` |
|---|---|---|
| Объектная база | Общая с основным репо | Отдельная копия |
| Место на диске | Минимально (только файлы ветки) | Полная копия `.git/` |
| Скорость создания | Секунды | Зависит от размера репо |
| Видимость коммитов | Сразу, без fetch/push | Нужен push/fetch |
| Ветки | Одна ветка на один worktree | Независимо |
| Удалённая работа | Нет (нужен общий диск) | Да, через remote |

## Когда worktree

- Нужно срочно переключиться на hotfix, не трогая незакоммиченную работу.
- Нужно быстро сравнить два состояния репо без stash-танцев.
- Хочешь собрать release-ветку параллельно с разработкой.
- Репо большое и clone займёт много времени/места.

```bash
git worktree add -b hotfix/nav ../cascadia-hotfix main
```

## Когда clone

- Нужна полная изоляция: эксперимент, который может сломать конфиги.
- Работаешь с чужим репо или форком.
- Нужна независимая история и remote-конфигурация.
- CI/CD: каждый билд в свежем clone.

```bash
git clone https://github.com/org/cascadia.git cascadia-experiment
```

## Подвохи

- **В worktree нельзя открыть ветку, уже открытую в другом worktree.** Если `feature/grid` открыта в основном репо — `git worktree add` с `feature/grid` выдаст ошибку.
- **Clone не видит коммиты из основного репо** без push/pull. Worktree видит всё сразу — это и плюс, и потенциальный источник путаницы (незапушенные коммиты из одного worktree видны в другом).
- **Worktree надо чистить.** После `rm -rf ../cascadia-hotfix` остаётся "мёртвая" запись — почисти через `git worktree prune`.
