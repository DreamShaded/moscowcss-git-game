# `git cherry` — какие коммиты ещё не влиты в upstream

`git cherry` (не путать с `git cherry-pick`) показывает, какие коммиты из текущей ветки ещё не присутствуют в upstream-ветке. Это инструмент для проверки: "что из моей ветки уже попало в main, а что нет?"

Git определяет это не по хешу (он меняется при rebase), а по **patch-id** — хешу от самого изменения, без метаданных коммита.

## Как использовать

```bash
# Какие коммиты из текущей ветки не в main:
git cherry main

# С подробностями (показать сообщения коммитов):
git cherry -v main

# Какие коммиты из feature/forms нет в develop:
git cherry -v develop feature/forms
```

Вывод:
```
+ abc1234 fix: button hover state    # этого коммита нет в main (+ = нет)
- def5678 refactor: modal z-index    # этот уже есть (- = есть)
```

Знак `+` — коммит ещё не влит. Знак `-` — влит (возможно через cherry-pick или rebase, хеш другой, но патч тот же).

## Когда это нужно

- Проверить перед PR: все ли нужные коммиты попали в feature-ветку.
- После cherry-pick: убедиться, что конкретные правки перенесены в release-ветку.
- Криминалистика: выяснить, в какие ветки попало конкретное исправление.

## Подвохи

- **`git cherry` — не `git cherry-pick`.** Это разные команды. `cherry` только показывает, `cherry-pick` переносит коммиты.
- **Patch-id может не совпасть** если коммит был изменён при переносе (например, разрешался конфликт по-другому). В таком случае `git cherry` покажет `+` даже для семантически уже влитого коммита.
- **`git branch --contains <hash>`** — более простой способ проверить, есть ли конкретный коммит в ветке. Для одного коммита — удобнее.
