# `git bisect` — бинарный поиск сломавшего коммита

`git bisect` автоматически делает бинарный поиск по истории коммитов, чтобы найти тот, который сломал что-то. Вместо того чтобы вручную проверять каждый из 50 коммитов, bisect проверяет логарифмическое количество — для 50 коммитов это около 6 шагов.

Принцип: ты указываешь "хороший" коммит (там всё работало) и "плохой" (там сломано). Git выбирает коммит посередине, ты проверяешь — хороший или плохой — и так до нахождения виновника.

## Как использовать (ручной режим)

```bash
git bisect start                 # начать поиск
git bisect bad HEAD              # текущий коммит — плохой
git bisect good <old_hash>       # этот коммит — хороший (всё работало)

# Git переключает на коммит посередине.
# Ты проверяешь вручную, потом:
git bisect good                  # этот коммит нормальный
# или
git bisect bad                   # этот коммит тоже сломан

# Git продолжает делить. Повторяй до:
# "X is the first bad commit"

git bisect reset                 # ОБЯЗАТЕЛЬНО: вернуться на исходную ветку
```

## Автоматический режим: `bisect run`

Если есть скрипт или команда, которая возвращает 0 при "хорошем" коммите и ненулевой код при "плохом":

```bash
git bisect start
git bisect bad HEAD
git bisect good <old_hash>
git bisect run npm run test:hover    # автоматически до результата
git bisect reset                     # не забыть!
```

## Подвохи

- **`git bisect reset` обязателен.** Без него репо остаётся в состоянии bisecting: HEAD в detached state, и следующие команды будут вести себя неожиданно. Всегда завершай сессию через `reset`.
- **Пропуск коммита:** если коммит нельзя проверить (не собирается, сломан по другой причине) — `git bisect skip`. Git обойдёт его.
- **Bisect работает с detached HEAD.** Не делай коммиты во время bisect-сессии — они создадутся в никуда.
- **Скрипт для `bisect run`** должен возвращать 0 (хорошо), 1–127 кроме 125 (плохо), или 125 (skip). Подробнее — в [`bisect-run-exit-codes`](./bisect-run-exit-codes.md).
