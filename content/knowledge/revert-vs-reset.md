# `git revert` vs `git reset` — когда что использовать

Оба инструмента "отменяют" коммиты, но принципиально по-разному. Выбор между ними определяется одним вопросом: **ветка уже запушена и её видят другие?**

## Разница

**`git revert <hash>`** создаёт **новый коммит**, который отменяет эффект указанного коммита. История не переписывается — к ней добавляется "revert-коммит". Безопасно для shared веток.

**`git reset <hash>`** **перемещает указатель ветки** на другой коммит, убирая последующие коммиты из истории. Переписывает историю — небезопасно для shared веток.

## Когда revert

- Ветка запушена, её пуллили другие (`main`, `develop`, `release`).
- Нужно отменить коммит в прошлом, не трогая последующие.
- Важно сохранить аудит-след: "вот коммит с ошибкой, вот коммит с отменой".

```bash
git revert <hash>            # создать revert-коммит (откроет редактор)
git revert --no-edit <hash>  # без редактора, с автоматическим сообщением
git push                     # безопасно, история не переписана
```

## Когда reset

- Коммит только локальный, не запушен.
- Feature-ветка, которую пишешь только ты (можно force push).
- Нужно полностью убрать коммиты из истории без следа.

```bash
git reset --hard HEAD~1      # убрать последний коммит
git reset --soft HEAD~3      # убрать 3 коммита, изменения оставить в индексе
```

## Ловушка: revert merge-коммита

Для отмены merge-коммита нужен флаг `-m`. Подробнее: [`revert-merge-commits`](./revert-merge-commits.md).

## Подвохи

- **`git revert` может создать конфликты** если последующие коммиты зависят от отменяемого.
- **`git reset` в shared ветке** + `git push --force` уничтожает историю для всех. Это почти всегда ошибка.
- **Revert revert'а** восстанавливает оригинальные изменения. Иногда это нужно — например, временно откатить фичу, потом вернуть.
