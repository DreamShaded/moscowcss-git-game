# Submodule — указатель на конкретный коммит

Submodule в git — это не "вложенный репозиторий, который автоматически обновляется". Это **указатель на конкретный коммит** во внешнем репозитории. Родительский репо хранит не файлы сабмодуля, а хеш коммита, на который он должен указывать.

Понять это важно, потому что интуиция "сабмодуль подтягивает последнее" — ошибочная и приводит к багам.

## Как это устроено

В родительском репо сабмодуль представлен:
- Записью в `.gitmodules` (URL и путь)
- **Специальным объектом типа `commit`** в дереве — просто хеш коммита во внешнем репо

```bash
git ls-tree HEAD design-tokens    # покажет: 160000 commit <hash> design-tokens
```

`160000` — специальный мод, означающий "это сабмодуль, а не файл или директория".

## Как работает update

```bash
git submodule update --init              # инициализировать и checkout на записанный хеш
git submodule update --init --recursive  # рекурсивно для вложенных сабмодулей
```

Без `--remote` — checkout именно на тот коммит, который записан в родительском репо. Это детерминированно и безопасно.

## Подвохи

- **Сабмодуль в detached HEAD.** После `update` сабмодуль находится в состоянии detached HEAD — ни на какой ветке. Это нормально. Не делай коммиты в сабмодуле в этом состоянии — они потеряются.
- **Pointer ≠ последний коммит.** Если во внешнем репо появились новые коммиты — сабмодуль об этом не знает, пока кто-то не обновит pointer и не закоммитит это изменение в родительском репо.
- **CI падает на новом pointer.** Если кто-то закоммитил новый pointer на коммит, который есть только локально у него — CI не сможет его скачать. Сначала push во внешнее репо, потом обновление pointer.
- **`git clone` без `--recurse-submodules`** создаст пустые директории сабмодулей. Нужен отдельный `git submodule update --init`.
